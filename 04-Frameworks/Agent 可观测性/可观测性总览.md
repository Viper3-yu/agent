# Agent 可观测性总览

## 概述

Agent 是非确定性的：同样的输入可能产生不同的执行路径。可观测性（Observability）通过监控、追踪、调试工具和方法，帮助团队理解 Agent 的决策过程、定位性能瓶颈和排查故障根因。它是将 Agent 从"能跑"推向"可信赖"的基础能力。

## 评估维度

| 维度 | 指标 | 计算方式 | 参考值 |
|------|------|----------|--------|
| Token 消耗 | 总量 + 分步骤 token 数 | 按 API 响应累计 | 依场景而定 |
| 延迟 | 首 token 时间 (TTFT) + 总时间 | P50/P95/P99 分位统计 | TTFT < 500ms |
| 工具调用成功率 | 工具调用成功次数 / 总调用次数 | 成功返回 / 总请求数 | > 95% |
| 端到端任务完成率 | 成功完成的任务 / 总任务数 | 人工标注或自动判定 | > 90% |
| 成本归因 | 按步骤归因的 token/API 费用 | 每步骤调用计费累加 | 依预算而定 |

## 核心工具

| 工具 | 厂商 | 定位 | 特点 |
|------|------|------|------|
| LangSmith | LangChain | 追踪 + 评估 | LangChain 生态原生 |
| LangFuse | LangFuse | 开源可观测 | 自托管，多框架支持 |
| Arize Phoenix | Arize | 可观测 + 评估 | OpenTelemetry 兼容 |
| Braintrust | Braintrust | 评估 + 日志 | 专注评估工作流 |

## 使用方法

### 输入格式

```json
{
  "trace_id": "trace-abc123",
  "agent_id": "agent-001",
  "spans": [
    {
      "span_id": "span-1",
      "type": "llm_call",
      "input": "用户查询内容",
      "output": "模型回复",
      "tokens_in": 150,
      "tokens_out": 300,
      "latency_ms": 1200,
      "metadata": { "model": "gpt-4o", "temperature": 0 }
    }
  ]
}
```

### 输出格式

```json
{
  "trace_id": "trace-abc123",
  "summary": {
    "total_tokens": 450,
    "total_latency_ms": 3500,
    "total_cost_usd": 0.012,
    "tool_success_rate": 0.95,
    "task_completed": true
  },
  "per_step_metrics": [
    { "step": "retrieval", "tokens": 100, "latency_ms": 800 },
    { "step": "generation", "tokens": 350, "latency_ms": 2700 }
  ]
}
```

### 代码示例

```python
from langfuse import Langfuse

langfuse = Langfuse()

# 创建 trace
trace = langfuse.trace(name="agent-query", user_id="user-001")

# 记录 span
span = trace.span(name="llm-call", input={"query": "北京天气"})
span.end(output={"response": "北京明天晴"}, usage={"total": 450})

# 获取指标
metrics = trace.get_metrics()
print(f"总 token: {metrics.total_tokens}, 延迟: {metrics.latency_ms}ms")
```

## 适用场景

- Agent 开发阶段的性能调试和瓶颈定位
- 生产环境的实时监控和告警
- 多版本 Agent 的 A/B 对比分析
- Token 成本归因和预算管控
- 故障排查和根因分析

## 局限性

- Trace 采集本身会引入额外延迟和存储开销
- 跨服务、跨 Agent 的分布式追踪实现复杂
- 可观测性是被动式观察，无法主动验证 Agent 质量（需配合 [[Harness 总览|Harness]]）
- 不同工具的数据格式不统一，迁移成本高

## 与其他评估方法的对比

| 特性 | 可观测性 | Harness 评估 | 人工评审 |
|------|----------|-------------|----------|
| 驱动方式 | 被动观察 | 主动构造场景 | 主动抽样 |
| 环境 | 生产 | 开发/预发布 | 任意 |
| 时机 | 发布后 | 发布前 | 发布前后 |
| 覆盖度 | 全量流量 | 可控测试集 | 抽样 5-10% |
| 成本 | 低（基础设施） | 中（LLM 调用） | 高（人力） |
| 典型产出 | 告警、Trace、Dashboard | 评估报告、回归基线 | 标注数据、质量评分 |
