# 评测管线设计

## 概述

评测管线（Evaluation Pipeline）是一套自动化流程，用于系统性地衡量 AI Agent 的输出质量。它将主观的"好不好用"转化为可量化、可追踪、可比较的指标体系，涵盖数据集构建、多维度评估、评分聚合与常见陷阱，帮助团队建立可重复的 Agent 质量保障体系。

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Dataset    │───>│    Agent     │───>│   Output     │───>│   Metric     │───>│   Report     │
│ Preparation  │    │  Execution   │    │  Collection  │    │ Computation  │    │ Generation   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
      │                   │                  │                  │                  │
  Golden set         Run agent on        Capture raw         Calculate          Generate
  + Synthetic        each test case      responses +         scores per         dashboard +
  + Edge cases       + metadata          intermediate        dimension          comparison
                                        traces
```

## 评估维度

单一指标无法全面反映 Agent 能力，需要从六个维度综合评估：

```
                    ┌─────────────┐
                    │  Composite  │
                    │    Score    │
                    └──────┬──────┘
           ┌───────┬───────┼───────┬───────┬───────┐
           v       v       v       v       v       v
      Correct  Faithful  Relevant  Safe  Efficient  Robust
      (30%)    (20%)     (15%)    (15%)  (10%)     (10%)
```

| 维度 | 指标 | 计算方式 | 参考值 |
|------|------|----------|--------|
| **Correctness** | 任务完成度、答案准确性 | exact_match, F1, task_success_rate | > 85% |
| **Faithfulness** | 输出是否忠于检索到的上下文，无幻觉 | groundedness_score, citation_accuracy | > 80% |
| **Relevance** | 回复是否匹配用户意图 | intent_match, relevance_score | > 75% |
| **Safety** | 是否产生有害输出、遵守 guardrails | toxicity_rate, policy_violation_count | 0% violation |
| **Efficiency** | 延迟、token 用量、成本 | p95_latency, avg_tokens, cost_per_query | 依场景而定 |
| **Robustness** | 边界用例处理、优雅降级 | edge_case_pass_rate, error_recovery_rate | > 85% |

### 代码示例

```python
from deepeval import evaluate
from deepeval.metrics import (
    AnswerRelevancyMetric,
    FaithfulnessMetric,
    HallucinationMetric,
)
from deepeval.test_case import LLMTestCase
from deepeval.dataset import EvaluationDataset

# 1. 定义指标
relevancy = AnswerRelevancyMetric(threshold=0.7)
faithfulness = FaithfulnessMetric(threshold=0.8)
hallucination = HallucinationMetric(threshold=0.3)

# 2. 构建测试用例
test_cases = [
    LLMTestCase(
        input="公司的退款政策是什么？",
        actual_output=agent.run("公司的退款政策是什么？"),
        expected_output="30天内无理由退款，需保留原始包装。",
        retrieval_context=[retrieved_docs],
    )
    for query, retrieved_docs, expected in dataset
]

# 3. 执行评测
dataset = EvaluationDataset(test_cases=test_cases)
results = evaluate(
    dataset,
    [relevancy, faithfulness, hallucination],
)

# 4. 生成报告
print(f"总用例: {len(test_cases)}")
print(f"通过率: {results.overall_metric_results.pass_rate:.1%}")
print(f"平均延迟: {results.avg_latency_ms:.0f}ms")
```

### 评估方法

#### Rule-based Evaluation

精确、低成本，适用于有明确标准答案的场景：

```python
def exact_match(pred: str, gold: str) -> bool:
    return pred.strip().lower() == gold.strip().lower()

def regex_match(pattern: str, pred: str) -> bool:
    return bool(re.search(pattern, pred, re.IGNORECASE))

def contains_all(pred: str, keywords: list[str]) -> bool:
    return all(kw.lower() in pred.lower() for kw in keywords)
```

#### LLM-as-Judge

使用强模型（GPT-4o / Claude）评估开放性输出：

```
Judge Prompt 模板:
┌─────────────────────────────────────────────────────┐
│ 你是一个严格的质量评估专家。                           │
│                                                     │
│ 用户问题: {query}                                    │
│ Agent 回复: {response}                               │
│ 参考答案: {reference}                                 │
│                                                     │
│ 请从以下维度打分 (1-5):                               │
│ 1. 准确性：回复是否正确                                │
│ 2. 完整性：是否覆盖了关键信息                          │
│ 3. 有用性：是否真正帮助了用户                          │
│                                                     │
│ 输出 JSON: { "accuracy": N, "completeness": N,      │
│              "helpfulness": N, "reasoning": "..." }  │
└─────────────────────────────────────────────────────┘
```

#### Hybrid Approach（推荐）

```
Hybrid Evaluation 流程:
                                          
  Agent Output ──> LLM Judge (100%) ──> Auto Score
                      │
                      ├── High Confidence (>0.9) ──> Accept
                      │
                      ├── Medium Confidence ──> Human Spot Check (10%)
                      │
                      └── Low Confidence (<0.5) ──> Force Human Review
```

### 管线架构模式

#### Sequential Pipeline

```
Input ──> [Preprocess] ──> [Agent Run] ──> [Collect] ──> [Evaluate] ──> [Report]
```

简单直接，适合小规模评测。瓶颈：串行执行慢。

#### Parallel Pipeline

```
                    ┌─> Evaluator A (correctness) ─┐
                    │                               │
Input ──> Agent ───┼─> Evaluator B (safety)     ───┼──> Aggregator ──> Report
                    │                               │
                    └─> Evaluator C (efficiency)  ──┘
```

多个评估器并行执行，显著缩短评测时间。

#### Hierarchical Pipeline

```
All Cases ──> [Fast Filter: rule-based] ──> Pass (>0.8) ──> Quick Report
                    │
                    └── Fail / Uncertain ──> [Deep Eval: LLM Judge] ──> Detailed Report
```

先粗筛再精评，平衡成本与覆盖度。

### 评分与聚合

#### Weighted Composite Score

```
composite = w_correct * S_correct
          + w_faithful * S_faithful
          + w_relevant * S_relevant
          + w_safe * S_safe
          + w_efficient * S_efficient
          + w_robust * S_robust

示例 (满分 1.0):
  composite = 0.30*0.92 + 0.20*0.88 + 0.15*0.95
            + 0.15*0.99 + 0.10*0.78 + 0.10*0.85
            = 0.898
```

#### Pass/Fail Thresholds

为每个维度设置最低门槛，任一维度不达标则整体判 fail：

```
Gate Rules:
  correctness  >= 0.85   (必须)
  safety       >= 0.95   (必须，零容忍)
  faithfulness >= 0.80   (必须)
  composite    >= 0.85   (必须)
```

#### Statistical Significance

A/B 比较时需检验差异是否显著：

- 使用 bootstrap resampling 计算 confidence interval
- 对于分类指标使用 McNemar's test
- 对于连续指标使用 paired t-test 或 Wilcoxon signed-rank test
- 样本量不足时结果不可靠，标注为"参考性"

## 适用场景

- RAG 系统的端到端质量评估
- Agent 版本迭代的回归测试
- Prompt 优化效果的量化对比
- CI/CD 流水线中的质量门禁
- 模型切换（如 GPT-4o 到 Claude）前后的性能对比

## 局限性

- 数据污染风险：测试数据泄露到训练集会导致评分虚高
- 指标博弈：优化单一指标而非真实质量
- LLM-as-Judge 存在评估偏差（对自身输出打分偏高）
- 评测成本随用例规模线性增长
- 快照陈旧：Agent 更新后评测集未同步会导致结果失真

## 与其他评估方法的对比

| 特性 | 评测管线 | 基准测试 (Benchmark) | 人工评审 | 在线 A/B 测试 |
|------|---------|---------------------|----------|-------------|
| 灵活性 | 高（自定义维度） | 低（固定任务集） | 高 | 中 |
| 成本 | 中 | 低 | 高 | 高 |
| 速度 | 快（分钟级） | 快 | 慢（天级） | 慢（周级） |
| 可复现性 | 高 | 高 | 低 | 中 |
| 覆盖度 | 可控 | 固定 | 抽样 | 全量 |
| 适用阶段 | 开发 + CI | 研究/基准 | 发布前校准 | 线上验证 |
