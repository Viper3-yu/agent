# 微调总览

## 概述

微调（Fine-tuning）是在预训练模型基础上，用特定领域数据进一步训练以适应特定任务的技术。它是 Prompt 优化、Few-Shot、RAG 之后更深层次的模型定制手段。微调可以显著提升模型在特定任务上的表现，同时减少推理时的 token 消耗。

## 核心原理

微调的核心思想是利用预训练模型已学到的通用知识，通过在特定任务数据上继续训练来调整模型参数。

```
预训练模型 (通用知识) + 特定任务数据 → 微调模型 (领域专精)

损失函数: L = -Σ log P(y_t | y_<t, x; θ)
其中 θ 为模型参数，全量微调更新所有 θ，参数高效方法（如 LoRA）只更新部分参数。
```

微调方法的选择取决于算力预算和数据规模：

| 方法 | 说明 | 适用 |
|------|------|------|
| 全量微调 | 更新所有参数 | 大数据+大算力 |
| [[LoRA 与 QLoRA\|LoRA]] | 低秩适配，只训练小矩阵 | 最常用，平衡效果和成本 |
| [[LoRA 与 QLoRA\|QLoRA]] | 量化 + LoRA | 消费级 GPU 也能微调 |
| RLHF | 人类反馈强化学习 | 对齐人类偏好 |
| DPO | 直接偏好优化 | RLHF 的简化替代 |
| ORPO | 偏好对齐（无需 SFT） | 最新方法 |

## 前置条件

### 数据要求

| 项目 | 要求 |
|------|------|
| 数据量 | SFT 通常需要 100-10,000 条高质量样本；RLHF/DPO 需要偏好对比数据 |
| 数据格式 | instruction-input-output 三元组（Alpaca 格式）或 conversation 格式 |
| 数据质量 | 高质量 > 高数量；低质量数据会导致模型退化 |

### 硬件需求

| 项目 | 最低配置 | 推荐配置 |
|------|----------|----------|
| GPU | 消费级 GPU（RTX 3090 24GB，配合 QLoRA） | A100 80GB / H100 |
| 显存 | 16GB（7B 模型 + QLoRA） | 80GB+（全量微调大模型） |
| 内存 | 32GB | 64GB+ |

## 什么时候该微调？

```
Prompt 优化 → Few-Shot → RAG → 微调
每一步复杂度递增，只在前一步不够时才进入下一步。
```

**适合微调的场景：**
- 需要特定输出格式（且 Prompt 难以稳定控制）
- 领域专业术语/风格（法律、医疗、金融）
- 减少 token 消耗（微调后 Prompt 可以更短）
- 提升小模型在特定任务上的表现

**不适合微调的场景：**
- 知识更新（用 RAG 更好）
- 通用能力提升（换个更好的模型更直接）
- 数据量不足（< 100 条高质量样本）

## 训练流程

### 步骤一：数据准备

```python
# 数据格式示例（Alpaca 格式）
import json

def prepare_dataset(data_path):
    with open(data_path, 'r', encoding='utf-8') as f:
        raw_data = json.load(f)

    formatted = []
    for item in raw_data:
        formatted.append({
            "instruction": item["instruction"],
            "input": item.get("input", ""),
            "output": item["output"]
        })
    return formatted
```

### 步骤二：配置训练参数

```python
from transformers import TrainingArguments

training_args = TrainingArguments(
    output_dir="./output",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,
    learning_rate=2e-5,
    warmup_steps=100,
    logging_steps=10,
    save_strategy="steps",
    save_steps=500,
    fp16=True,  # 混合精度训练
)
```

### 步骤三：启动训练

```bash
# 使用 Hugging Face Transformers
python train.py --config config.yaml

# 使用 Axolotl
axolotl train config.yml

# 使用 Unsloth（加速）
python train_unsloth.py
```

### 步骤四：评估与部署

```python
# 评估方法
from transformers import pipeline

# 加载微调后的模型
model_path = "./output/checkpoint-final"
pipe = pipeline("text-generation", model=model_path)

# 测试推理
result = pipe("请用一句话总结微调的作用：", max_new_tokens=100)
print(result[0]["generated_text"])

# 部署选项：合并权重后导出为 GGUF/ONNX/SafeTensors
```

## 关键超参数

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| learning_rate | 2e-5 ~ 5e-5 | 全量微调；LoRA 可用 1e-4 ~ 3e-4 |
| num_epochs | 1-3 | 数据量少时增加 epoch |
| batch_size | 4-16 | 根据显存调整 |
| warmup_ratio | 0.03-0.1 | 学习率预热比例 |
| weight_decay | 0.01 | 权重衰减，防过拟合 |
| max_seq_length | 512-4096 | 根据任务和显存设置 |

## 与其他方法对比

| 特性 | 微调 | RAG | Prompt Engineering |
|------|------|-----|-------------------|
| 实施成本 | 高（需要 GPU 和数据） | 中（需要向量数据库） | 低（仅需 Prompt） |
| 知识更新 | 需重新训练 | 实时更新 | 无需更新 |
| 推理延迟 | 与基座相同 | 略增（检索开销） | Prompt 更长 |
| 效果上限 | 最高 | 中等 | 受限于 Prompt 长度 |
| 适用场景 | 领域专精、格式控制 | 知识密集型 | 快速原型 |

## 常见问题与排坑

- **过拟合**：数据量少时容易过拟合，增加 dropout 或减少 epoch
- **灾难性遗忘**：微调后通用能力下降，可用 replay 数据或降低学习率
- **数据质量 > 数据数量**：100 条高质量数据优于 10,000 条低质量数据
- **评估指标选择**：不要只看 loss，要用下游任务的实际指标评估
- **显存不足**：使用 gradient checkpointing、混合精度、或 QLoRA
